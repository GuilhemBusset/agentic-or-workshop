# Paradigm Shift Script: From Code Output to Research Impact

## Purpose
Instructor narrative for explaining the shift in software value in the era of coding agents, mapped directly to optimization research practice.

## Narrative Spine

### Statement 1: Code is cheap.
Instructor line:
"Generating code is now low-friction. Draft implementations can be produced quickly by both humans and coding agents."

Supporting point:
- The bottleneck has moved away from typing syntax.
- Baseline scripts, boilerplate, and first-pass implementations are commoditized.

OR classroom example:
- A student can generate a transportation model skeleton in minutes.
- The presence of code does not imply the model answers the right question.

Transition prompt:
"If code is easy to produce, what becomes hard and valuable?"

### Statement 2: High-quality code is expensive.
Instructor line:
"Reliable code still requires careful design, testing, and iteration."

Supporting point:
- Quality requires explicit constraints, edge-case handling, and reproducible checks.
- Agents can accelerate implementation, but verification effort remains substantial.

OR classroom example:
- Two formulations may both run, but only one handles infeasibility diagnostics and data anomalies correctly.
- Quality appears in robustness, not just execution success.

Transition prompt:
"If robust code is expensive, what is even more valuable than robust code?"

### Statement 3: Great solutions are even more expensive.
Instructor line:
"The highest value is not code quality alone; it is decision quality and measurable impact."

Supporting point:
- Great solutions combine technical validity with stakeholder fit and operational feasibility.
- They include assumptions, tradeoffs, and communication tailored to decision-makers.

OR classroom example:
- A technically correct network design may fail adoption if assumptions are opaque or fairness tradeoffs are unexplained.
- A great solution includes sensitivity analysis and a clear recommendation narrative.

## OR Mapping: What Becomes the Core Differentiator

### 1) Problem Framing Quality
- Are we solving the right decision problem?
- Is objective choice aligned with organizational intent?
- Are constraints representing real operational limits?

Instructor cue:
"A fast wrong model is still wrong."

### 2) Assumption Quality and Traceability
- Are assumptions explicit, testable, and documented?
- Can we trace each modeling decision to data or policy rationale?
- Do we distinguish facts, estimates, and guesses?

Instructor cue:
"If you cannot trace an assumption, you cannot defend the recommendation."

### 3) Experimental Design and Evaluation Quality
- Are scenario tests representative of real uncertainty?
- Are stress cases included, not just happy-path cases?
- Is comparison against baselines fair and reproducible?

Instructor cue:
"Great modeling is inseparable from great evaluation design."

### 4) Decision Impact and Stakeholder Communication
- Is the recommendation understandable to non-modelers?
- Are tradeoffs and limitations communicated clearly?
- Is there an implementation path that decision-makers can trust?

Instructor cue:
"If stakeholders cannot act on it, the solution is incomplete."

## Teaching Sequence (Suggested Delivery)
1. Start with a quick poll: "How long to generate a first model draft now vs two years ago?"
2. Deliver the three statements with one OR example each.
3. Run a short discussion on where students currently spend effort.
4. Re-anchor effort allocation:
- less effort on boilerplate code generation,
- more effort on framing, assumptions, evaluation, and communication.

## Close
Key message to students:
"Coding agents compress the cost of producing code. Your research advantage shifts to designing the right problem, validating the right evidence, and communicating the right decision."
